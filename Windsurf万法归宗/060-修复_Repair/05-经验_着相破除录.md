## 真本源识别六要

### 一·静观全相不预设

不带"必是 token / 必是代理 / 必是 ..."之先入预设·遍 6 层 (现象/UI/ext-host/ACP/内核/二进制) 取实证。

### 二·log 是真口

不信主观判·只信 log。`Windsurf ACP.log` / `WAM.log` / `Windsurf.log` / `exthost.log` / `renderer.log` / `main.log` —— 6 路并察 · 真因自显。

### 三·net stat 是真证

`Get-NetTCPConnection -State Established` 看真出向。若网真通官方 + log 仍显失败·则失败在 ext-host 内部 (而非网络)。若网未通官方·则失败在网络/代理/反代层。

### 四·进程关系真测

`Get-CimInstance Win32_Process` + 看 ParentProcessId / CommandLine。ghost 进程之识别靠此。

### 五·二进制层勿避

若上 5 层皆查无问题但症状仍在·必有二进制层之硬编/打补丁/魔改。`grep` 之 `dist/extension.js` / `Windsurf.exe` / 内核 binary 内之可疑字符串 (反代端口 / localhost URL / 魔改之 marker)。

### 六·备份件是真本源指纹

`*.bak_predao_*` / `*.bak_pre*` / `*.original` / `*.bak.*` —— 这些 backup 件往往是反代/魔改体系自身留下的指纹。**反代体系替换文件前自备改前·后改之**。这就是反代之"血印"。

主公电脑 binary 之一切非官方下载之 .bak 件·都暗示某种被改痕迹。

## 治本之三诫

### 诫一·勿替整 Windsurf 装

不建议重装 Windsurf 整体·因：
- 主公 user-data-dir 之 workspaceStorage / extensions / settings 是宝
- 重装会删 WAM (devaid.rt-flow-2.7.5) + accounts.md 等关键扩展状态
- 主公个人 keybindings / snippets / themes 会失

应只针对性还原**被改之单一文件** (extension.js)。

### 诫二·还原必备份当前

```powershell
Copy-Item $extJs ($extJs + '.bak_8878hack_' + (Get-Date).ToString('yyyyMMdd_HHmmss'))
```
若新版反生异 (主公装的是与 bak_predao 不同 codeiumVersion 的)·可即时回滚。

但实际：bak_predao_1778654701851 与 5/8 装的 codeiumVersion 1.48.2 之 product.json 兼容·无回滚需。

### 诫三·治本后必让 WAM 自注·勿手注

WAM ext 自有 `auto-verify` + `provideAuthTokenToAuthProvider` 之自注流。

主公无需：
- 手动调 self-serve.windsurf.com 之 Login API
- 手动写 windsurf-auth.json
- 手动注 token 到 secret storage

主公只需：杀全 + 还原 + 净启 + 等 90s · WAM 自然成事。
