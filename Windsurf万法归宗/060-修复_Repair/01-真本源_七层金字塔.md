## 各层证据

### Layer 1 · 主公图证 (现象)

主公 RDP 中 Windsurf UI 真态：
- Windsurf 主窗口已显
- Cascade panel 已 mount (右侧 W 标志 · "Cascade Code")
- 历史 chat 列表显 ("New Cascade 3d / Explaining Dao Fa Zi Ran 5d / ...")
- 底栏 `D100%·W100% 32/41号` (WAM 实时显号活态)
- 输入框可见: `Ask anything (Ctrl+L)`

**但主公说"无任何变化"** — 即输入 prompt + 发送·**chat 真无响应**。

### Layer 2 · Cascade panel 启序异

`renderer.log`:
```
[warning] IWorkbenchContributionsRegistry#getContribution('windsurf.cascadePanel'):
          contribution instantiated before LifecyclePhase.Restored!
```

panel UI 已 mount·但其后端依赖 ext-host 之 codeium.windsurf 服务·该服务之 ACP 子系统死循环 (见 Layer 4)·导致 chat send 无后端响应。

### Layer 3 · WAM 与 ext-host 异象

### Layer 4 · ACP 死循环 (病灶)

`Windsurf ACP.log`:
```

[23:24:01] Error loading local registry: EntryNotFound (ENOENT)

           dial tcp 127.0.0.1:8878
```

**关键**：`devinCloud=true` 表示 Cascade chat 之主路径走 devin-cloud agent·但 agent 之 registry fetch 死 → 无 agent 可注 → chat 无后端可响。

### Layer 5 · Windsurf 内核之参数 (真因前)

extension.js 内启 language_server 子进程的代码 (当前·被改后)：
```javascript
o = ["--api_server_url", "http://127.0.0.1:8878",
     "--run_child", "--enable_lsp",
     "--extension_server_port", A.extensionServerPort.toString(),
     "--ide_name", (0,N.getWindsurfIdeName)()];
...
o.push("--inference_api_server_url", "http://127.0.0.1:8878")
```

language_server 启时被强制指向 :8878 反代·但 :8878 反代进程已 5/15 主公自损·无监听 → 所有依赖 :8878 的子系统死。

## 反代体系之路径回溯

历史上的"dao 系"反代体系 (universal_relay.js / dao-yi-shun-tian / dao-yi-mu / 等) 在 Windsurf 装后修改 extension.js·令其启 language_server 时硬编 :8878 作为 API server·此样反代进程能截/中转所有请求。

22:45 主公自损反代体系时·只清了：
- 反代进程 (universal_relay.js 之 node 进程)
- 配置层 (settings.json 内 dao.* keys)
- 端口 (8957/11435 等绑定释放)

**未清此 binary 层之 +9 字节** — 此即「前识者·道之华也」之实证。

ACP fetchRegistry 始终找 :8878·进死循环·Cascade chat 之 devin-cloud agent 无 registry 可注。
