## 一 · 真本源之"一"——主公一 Windsurf 账号

### 1.1 ~/.wam/wam-state.json (主公真本源之入 · 真 schema)

```json
{
  "version": "2.x",
  "savedAt": "ISO8601",
  "health": { ... },
  "blacklist": [ ... ],
  "switches": [ ... ],
  "changesDetected": [ ... ],
  "activeEmail": "jitneusey4@gmail.com",
  "lastInjectPath": "...",
  "activeApiKey": "devin-session-token$xxx",  // ★ vm_omni.js 直读此
  "activeTokenShort": "...",
  "activeApiServerUrl": "...",
  "lastRotateAt": "ISO8601",
  "accountMeta": { ... }   // ★ 多账号元数据 (非 accounts[] 数组)
}
```

**位置**: `%USERPROFILE%\.wam\wam-state.json` (Windows · `~/.wam/wam-state.json` Linux/Mac)

**主公无须动**: vm_omni.js 之 `loadToken()` 自读 `activeApiKey` (若以 `devin-session-token$` 起则自动 strip 此前缀) · 即可 spawn 一 Devin VM。

**主公 WAM 切换管理器 (附图 · 182 号)**: 主公可在 GUI 内切活账 · vm_omni.js 启动瞬即取 `activeApiKey`。

### 1.2 主公多 Windsurf 账号 (一 → 二 → 三 · 真有 182 号)

主公之 WAM 切换管理器持 **182 号** Windsurf 账号 (附图: 93 可用 · 88 耗尽 · 1 未验)。每号 D 重置 1.1h · W 重置 1.1h · 满血账可即用。

**法 1: 通过 WAM GUI 切活账 (推 · 不动 cli)**

主公在 WAM 切换管理器内点选某 D100/W100 之号 → `activeApiKey` 自动更新 → 此后 `node vm_omni.js` 即用此号 spawn VM。

**法 2: cli 显式覆 (印 101 · 不污 wam-state)**

```pwsh
# vm_omni.js 之 --token 参 (直传 · 优先于 WAM)
node ..\00_本源\vm_omni.js --token 'devin-session-token$xxx'

# 或走 env (vm_omni.js 之 fallback 路 · 见 loadToken)
# 注: vm_omni.js 之 token 参非 env 名 · 主公诚需 cli 显式
```

**法 3: spawn 多 VM (主公手切 WAM 之号 + 多次 spawn)**

```pwsh
# 主公 WAM 切号 A → spawn VM1
node ..\00_本源\vm_omni.js
# 主公 WAM 切号 B → spawn VM2
node ..\00_本源\vm_omni.js
# 主公 WAM 切号 C → spawn VM3
node ..\00_本源\vm_omni.js
```

**典**: 三号 → 三 VM → 公网三 omni-router URL → 主公任意分流 (e.g. 一 prod / 一 dev / 一 sandbox)

**注意 · 1 号 1 VM/24h 之实**: WAM 之同一账号若已 spawn 过一件 VM 在 24h 内 · 再 spawn 可能复用同 session (vm_omni 之 `--session devin-xxx` 显指可升级)。主公若要**N 件真独立 VM** · 须 spawn 时切 N 个不同 WAM 账号。

**法 4 · spawn_N.ps1 任 N 并发 (印 125 · 推到极)**

主公 `tokens_dao_123.txt` 持 64 个 token · 一字 -Count 64 即起 64 件 VM (一一对一)：

```pwsh
# 0 ACU 预览 · 看会做什么
.\spawn_N.ps1 -Count 64 -Action preview

# ~64 ACU 真起 · 主公真金真银承
.\spawn_N.ps1 -Count 64 -Action go

# 等齐起 (~10min)
.\spawn_N.ps1 -Action wait

# 自动 vm_proxy_deploy 每件 (含 per-VM SP 注入)
.\spawn_N.ps1 -Action deploy
```

**典 · -Count 7 (7 SP 态循环)**: i=0→bypass · i=1→override · ... i=6→usernote · 一一对应 7 个 token · 各 VM 独立 SP 态。

**典 · -Count 64 (主公真池满)**: 64 件 VM 一一对一 64 个 windsurf 账号 · per-VM SP 隔离 · 主公 client 端可负载均衡分流。

**主公诏极**: -Count 178 (主公 WAM 之 178 账号上限 · 须先备 178 token 至 -TokenFile)

## 三 · token 之"三守" (道义)

```text
✓ ① 仅本机本用户读
   tokens_*.txt 之 ACL (NTFS)：仅当前用户 · 不可 share
   ~/.wam/wam-state.json 同

✓ ② 仅 VM 内之 dao_proxy 进程读
   vm_proxy_deploy.js 之上传走 omni file API (Devin 官口) · 加密传输
   VM 内 chmod 600 (read-only by user)

✓ ③ 不外传 · 不日志全文
   dao_proxy 之 logger 仅记 last4 (e.g. "...a1b2") · 不全文
   /metrics endpoint 仅返计数 · 不返 token
   /health 仅返 pool size · 不返 token list
```

**主公**: 切勿 git commit `tokens_*.txt`！(已在 `.gitignore` 内)

### 4.1 号源三处之合 (道一)

主公之 windsurf 号散在三处 · 印 127 之 `vm_dao_unify` 合为一表:

```text
  ~/.dao/accounts.json    ── ~124 号 (主公全号库)
  ~/.wam/wam-state.json   ── 现活之 activeApiKey + accountMeta
  tokens_dao_123.txt      ── 64 个 Devin token (本机)
  tokens_ws_59.txt        ── 59 个 Windsurf token (本机)
           ↓
  vm_dao_unify  ─────  decode JWT exp · 去重 · 出一表
           ↓
  71 号一表: fresh │ used │ dead
```

### 4.2 主公一字借桥 (印 128)

```pwsh
# 路 1 · 一笔便活.ps1 借桥
.\一笔便活.ps1 unify-list                                # 默 list 全 71
.\一笔便活.ps1 unify-list -BridgeArgs '--status','fresh'  # 仅 fresh

# 路 2 · 直调桥 (主公亲接入 ../00_本源/_VM底层桥/)
cd ..\00_本源\_VM底层桥
.\vm_unify.cmd list                                 # 全表
.\vm_unify.cmd list --status fresh                  # fresh 仅
.\vm_unify.cmd pick --count 4 --status fresh        # 出最优 4 号 (JSON)
.\vm_unify.cmd decode --tag c7d9bf15                # decode JWT
.\vm_unify.cmd refresh                              # 重 unify
```

### 4.3 状 · fresh / used / dead 之道义

| 状 | 何为 | 宜用 |
|----|------|------|
| **fresh** | JWT exp 足 + 未起过 VM 在 24h 内 | **★ 首选** · spawn 新 VM (一号一 VM) |
| **used** | JWT exp 足 但已起过 VM (1 号 1 VM/24h) | 仅探实现状 · 不可重 spawn |
| **dead** | JWT exp 不足 / refresh failed | 不可用 · 主公可 OAuth 重 |

### 4.4 道义守 (印 128 · 鸡犬相闻)

```text
✓ 不复制   · vm_unify 件居虚拟机资源 · 反代仅借桥
✓ 不偏 token · unify 只读元 · 不外传
✓ 不走 ACU · list/decode/refresh 均 0 ACU
✓ 道并行   · 主公仍可独立 cd 虚拟机资源/ 直用 vm_dao_unify.cmd
```

### 5.1 何为「1:1 持久化」

**与 spawn_N.ps1 之分**:

| 维度 | spawn_N.ps1 (印 125) | 一号一VM.ps1 (印 128) |
|------|---------------------|----------------------|
| token 源 | tokens_dao_123.txt 任行 | vm_unify pick --status fresh (真活号) |
| 持久 | _state/spawn_N_state.json (临时) | **bindings.json (主公真据)** |
| 重起 | 每次即新 (无关联) | 同号只 1 VM · idempotent |
| 解绑 | 无概念 | 显 `unbind` 留 history (守玄德) |
| 适用 | 试验 / 压测 | **生产 / 长跑** |

### 5.2 主公一号一日

```pwsh
cd ..\06_号VM绑定

# ① 看现状 (已绑定几件)
.\一号一VM.ps1 status

# ② 计 · 0 ACU · 看会选哪 N 号
.\一号一VM.ps1 plan -Count 4

# ③ 真起 · ~4 ACU
.\一号一VM.ps1 go -Count 4

# ④ 等齐起
.\一号一VM.ps1 wait

# ⑤ 验活
.\一号一VM.ps1 verify

# ⑥ 主公自配某号之 SP 态
.\一号一VM.ps1 set-sp -Idx 0 -SpStrategy dao -SpDaoChapter 22

# ⑦ 某号死·同号重起续绑
.\一号一VM.ps1 rebind -Idx 0

# ⑧ 主公主动解绑·留 history
.\一号一VM.ps1 unbind -Idx 0

# ⑨ 导出 bindings.json (主公备份)
.\一号一VM.ps1 export > bindings_$(Get-Date -Format yyyyMMdd).json
```
